# PRD: AiPlayMusic MVP — AI 原生音乐播放器

## Problem Statement

现有音乐播放器的核心矛盾在于：用户经常处于"想听点什么但说不清楚"的模糊情绪状态（如"适合周五晚上下班开车的歌"），但现有产品只提供两种路径——要么是过度依赖历史记录的推荐算法（信息茧房），要么是要求用户提供精确关键词的搜索框。两者都无法服务"知道想要的 Vibe，但不知道搜什么"的场景。

## Solution

AiPlayMusic 是一个以 AI 对话为核心交互的 Web 音乐播放器 Demo。用户通过自然语言或点击情境卡片表达模糊的情绪诉求，AI 通过最多 3 轮共情对话理解用户意图，最终从精选曲库中推荐 3-5 首歌曲，以精美的音乐实体卡片呈现在聊天流中，用户可直接点击试听，完成从"情绪输入"到"听觉满足"的完整闭环。

**产品定位**: Agentic UI Showcase / Demo，展示"下一代 AI 产品交互范式"。

**目标受众**: AI 行业的面试官、开发者和产品经理同行。

**用户画像**: 追求情绪价值、容易产生听觉疲劳的年轻数字原生代（Gen Z）。

## User Stories

### 核心体验

1. As a 用户, I want to 在首页看到一个醒目的 AI 唤醒按钮, so that 我不需要思考就能开始互动。
2. As a 用户, I want to 点击唤醒按钮后看到 AI 主动打招呼并展示 3-4 个精美的情境选项卡片, so that 我可以零输入地开始表达情绪。
3. As a 用户, I want to 直接点击一个情境卡片（如"深夜 emo"）, so that 我不需要打字就能告诉 AI 我的心情。
4. As a 用户, I want to 在输入框中用自然语言描述我的模糊需求（如"适合雨天发呆的歌"）, so that AI 能理解我说不清楚的情绪。
5. As a 用户, I want to AI 用共情话术回复我（而不是冷冰冰的列表）, so that 我感受到被理解的情绪价值。
6. As a 用户, I want to 在最多 3 轮对话后看到 AI 的推荐结果, so that 我不会因为无休止的追问而失去耐心。
7. As a 用户, I want to 推荐结果以精美的音乐实体卡片呈现在聊天流中（封面、歌名、艺人、推荐理由）, so that 推荐体验是沉浸式的而非干巴巴的文字列表。
8. As a 用户, I want to 直接点击音乐卡片上的播放按钮立即试听, so that 我能即时验证 AI 的推荐是否符合我的心情。
9. As a 用户, I want to 点击"换一批（Not My Vibe）"按钮, so that AI 重新推荐另外一组歌曲而不需要我重新描述需求。
10. As a 用户, I want to 当 AI 推荐的歌曲在播放时底部播放条同步显示当前曲目和进度, so that 我能控制播放。

### AI 引擎配置

11. As a 用户, I want to 通过设置入口选择不同的 AI 厂商（DeepSeek、Kimi、智谱等）, so that 我可以使用自己偏好或已有额度的 AI 服务。
12. As a 用户, I want to 只需要填写 API Key 并选择模型名即可完成配置, so that 配置过程简单快捷。
13. As a 用户, I want to 配置信息在浏览器中持久保存, so that 我下次打开不用重新配置。

### 异常兜底

14. As a 用户, I want to 在没有配置 API Key 时点击 AI 按钮能看到友好的引导提示, so that 我知道需要先去配置而不是看到报错。
15. As a 用户, I want to 在 API 调用超时或网络断开时看到优雅的错误提示和重试按钮, so that 我不会以为应用崩溃了。
16. As a 用户, I want to 即使 AI 返回了无法解析的结果系统也能优雅降级, so that 界面永远不会白屏。

### 会话管理

17. As a 用户, I want to 点击"重新开始"按钮清空当前对话, so that 我可以随时开启一轮全新的情绪破冰。

### 明确意图处理

18. As a 用户, I want to 当我提出明确的歌曲/歌手需求时 AI 直接在曲库中搜索匹配, so that 我不会被强制走完共情对话流程。
19. As a 用户, I want to 当曲库中没有我指定的歌曲时 AI 用高情商话术安抚并推荐风格最接近的替代, so that 我不会得到"找不到"的生硬拒绝。

## Implementation Decisions

### 模块 1：精选曲库（Music Catalog）

- 30 首免版权歌曲，覆盖 6 个情绪分区，每区 5 首
- 6 个情绪分区：极致燃向、深夜 emo、绝对专注、慵懒微醺、户外兜风、助眠放空
- 数据结构：`{ id, name, artist, coverUrl, audioUrl, duration, genreTags[], moodTags[], description }`
- 以静态 JSON 文件 + 本地音频文件的形式随项目托管
- 音频来源：Pixabay Music / Free Music Archive 等免版权平台
- 封面图：每首歌配一张高质量封面（可用 Unsplash 免版权图片）
- **AI 防幻觉核心机制**: 完整曲库清单注入 AI 的 system prompt，AI 输出结构化的歌曲 ID 列表，系统端通过 ID 匹配渲染卡片。AI 无法推荐曲库之外的任何歌曲。

### 模块 2：AI 对话引擎（Chat Engine）

- **多厂商支持策略**: 统一走 OpenAI 兼容协议（`/v1/chat/completions`）
  - 预置厂商列表，每个厂商 = 一个 `baseUrl` + 默认模型列表
  - 覆盖厂商（至少）: OpenAI、DeepSeek、Moonshot (Kimi)、智谱 (GLM)、通义 (Qwen)、Google Gemini
  - 用户选择厂商 → 填 API Key → 选/填模型名
  - 统一的 fetch 调用层，差异仅在 baseUrl 和 headers
- **System Prompt 设计**:
  - 注入完整曲库 JSON（ID + 名称 + 艺人 + 标签）
  - 强制 AI 用指定 JSON 格式输出推荐结果：`RECOMMENDATIONS: [{"id": "xxx", "reason": "xxx"}, ...]`
  - 内置角色定义：专业音乐推荐助手，共情话术风格
  - 内置对话轮次感知指令
- **对话控制逻辑（3 轮强制闭环）**:
  - 第 1 轮：AI 主动破冰 + 3-4 个情境选项卡片
  - 第 2 轮：AI 可追问细化（也可直接推荐）
  - 第 3 轮：无论如何必须输出最终推荐结果（3-5 首 + 推荐理由）
  - 如果用户第 1 轮就有明确目标，直接匹配曲库，跳过后续轮次
- **响应解析逻辑**:
  - 从 AI 文本回复中提取 `RECOMMENDATIONS: [...]` JSON 块
  - 按 ID 匹配曲库数据
  - 匹配失败的 ID 静默丢弃，不渲染
  - 如果全部匹配失败 → 降级为纯文本展示 + 错误提示
- **"换一批"逻辑**: 向 AI 追加一条系统消息，告知上一批被拒绝的歌曲 ID，要求重新推荐不同的歌曲

### 模块 3：AI 聊天 UI（Chat Panel）

- 基于现有 `AIChatPanel.tsx` 重构
- **破冰场景卡片组件**: 塔罗牌风格的精美卡片，每张代表一个情绪场景，带图标和简短描述
- **音乐实体卡片组件**: 在聊天流中直接渲染，包含：
  - 歌曲封面图（圆角）
  - 歌名 + 艺人
  - AI 的一句走心推荐理由
  - 播放/暂停按钮（点击即播放）
- **"Not My Vibe" 按钮**: 在推荐卡片区底部，点击后触发换一批逻辑
- **对话进度感知**: UI 层面不需要显式的进度条，但内部通过轮次计数控制 AI 行为

### 模块 4：音频播放器（Audio Player）

- 基于 HTML5 `<audio>` API 的全局单例播放器
- **音乐卡片交互**: 点击播放按钮 → 加载音频 → 立即播放；再次点击 → 暂停
- **底部 PlayerBar 同步**: 显示当前播放曲目的封面、歌名、艺人；播放/暂停按钮；进度条（可拖拽）
- **排他性播放**: 同时只能播放一首，点击另一首时自动切换
- **MVP 不做**: 上一首/下一首、循环模式、随机播放、音量控制（保留 UI 但不接入逻辑）

### 模块 5：AI 引擎配置（Provider Config）

- 设置入口：聊天面板头部的齿轮图标（沿用现有设计）
- 配置表单：
  - 厂商下拉菜单（预置列表，含 baseUrl）
  - API Key 密码输入框
  - 模型名下拉/输入框（根据厂商动态变化）
- 持久化：LocalStorage
- 配置变更后自动重新初始化对话引擎

### 模块 6：异常兜底与会话管理（Error & Context Management）

- **未配置 API Key**: 点击 AI 按钮后展示引导卡片而非空面板，引导用户前往设置
- **API 调用失败（超时/网络错误/鉴权失败）**: 
  - 友好的错误提示消息（区分"Key 无效"和"网络问题"）
  - 提供"重试"按钮
  - 鉴权失败时直接引导前往设置页
- **AI 输出解析失败（无法提取歌曲 ID）**: 
  - 将 AI 的原始文本作为普通聊天消息展示
  - 追加一条系统提示："推荐解析失败，请点击重试或换个说法"
  - 提供"重试"按钮
- **会话重置**: 聊天面板顶部提供"重新开始"按钮（扫帚/刷新图标），点击后清空所有对话历史，回到初始破冰状态

### UI 融合方案（基于现有 Spotify Clone 布局）

- **保留整体框架**: Sidebar + MainContent + PlayerBar 布局不变
- **TopBar**: "AI 推荐"按钮视觉升级为更醒目的"万能按钮"（保留渐变色 + Sparkles 图标）
- **AI 聊天面板**: 点击万能按钮后从右侧滑入（沿用现有 `isChatOpen` 模式），视觉全面升级
- **MainContent**: 保留现有货架浏览内容，作为非 AI 模式下的默认首页
- **PlayerBar**: 从纯静态 UI 升级为接入真实播放状态的功能组件

### 技术栈决策

- **框架**: 沿用 Next.js 15 (App Router) + React 19 + TypeScript
- **样式**: 沿用 Tailwind CSS 4
- **状态管理**: 引入轻量方案管理播放器状态和 AI 对话状态（React Context 或 Zustand，实现时决定）
- **AI 通信**: 浏览器端直接调用各厂商 API（无自建后端），走 OpenAI 兼容协议
- **音频**: HTML5 Audio API
- **动画**: 沿用 motion (Framer Motion)

## Testing Decisions

### 测试原则

- MVP 阶段追求极致交付速度，**放弃前端 UI 自动化测试**
- 自动化单元测试 **100% 聚焦于核心逻辑层**
- UI 层面完全依赖手动验收

### 必须测试的模块

1. **AI 响应解析器（Response Parser）**
   - 输入：AI 返回的原始文本
   - 输出：结构化的歌曲 ID + 推荐理由列表
   - 测试用例：正常输出、格式错乱、无 RECOMMENDATIONS 块、ID 不存在于曲库、空数组、混合有效/无效 ID

2. **对话轮次控制器（Turn Controller）**
   - 测试 3 轮阈值的硬性阻断逻辑
   - 测试第 1 轮明确意图的快速通道
   - 测试轮次计数在"换一批"和"重新开始"后的正确重置

3. **System Prompt 构建器（Prompt Builder）**
   - 测试曲库数据正确注入 prompt
   - 测试轮次信息正确注入 prompt
   - 测试"换一批"时排除列表正确注入 prompt

### 好的测试标准

- 只测试外部行为（输入 → 输出），不测试内部实现细节
- 每个测试用例独立，不依赖执行顺序
- 覆盖正常路径 + 所有已知的边界/异常路径

## Out of Scope

以下内容在本次 MVP 中**明确不做**：

- 接入全网海量曲库（网易云、QQ 音乐等第三方 API）
- 跨页面的全局播放控制条（MVP 仅单页应用）
- 用户账号体系、登录、注册、长期偏好记忆
- AI 回复的实时逐字打字动效（允许使用"AI 思考中"加载状态平替）
- 收藏歌曲、创建歌单等重度管理操作
- 上一首/下一首、循环播放、随机播放等完整播放器功能
- 移动端适配（MVP 仅桌面端）
- 国际化 / 多语言
- SEO 优化
- 部署与 CI/CD 流程
- Python Agent 后端服务（MVP 阶段 AI 通信完全在浏览器端完成）

## Further Notes

- **设计基调**: 暗黑高级感，与现有 Spotify Clone 的视觉语言一致。所有新增组件必须与现有 UI 风格无缝融合。
- **音频文件管理**: 免版权音频文件直接放在项目 `public/` 目录下，封面图同理。确保所有素材的版权合规性。
- **AI Prompt 工程是核心竞争力**: System Prompt 的设计质量直接决定产品体验，需要反复调优。建议将 prompt 模板独立管理，方便迭代。
- **"零幻觉"是铁律**: 任何绕过曲库 ID 匹配机制的推荐行为都是 Bug，必须在解析层彻底阻断。
- **项目结构演进方向**: 当前 MVP 全部在 `web/` 目录完成。后续如需 Python Agent 服务，按 `docs/ai_native_music_player_initial.md` 的规划拆分为 `web/` + `agent/` + `infra/`。
