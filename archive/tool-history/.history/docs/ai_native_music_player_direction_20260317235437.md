# 项目初始化说明

## 目标

这份文档只用于给后续的 opencode 提供**项目初始化方向**。  
当前阶段不讨论详细功能、产品交互、推荐策略或 agent 细节。

重点只解决：

- 本地开发环境怎么搭
- 项目目录大概怎么分
- 应该用什么环境管理方式
- 当前阶段不该过早做什么

---

## 当前结论

目前项目处于第一个版本这是一个 **Web + Agent** 的项目，建议按“双栈”方式初始化：

- Web 侧单独管理
- Python agent 侧单独管理
- 不要一开始把整个项目都放进 conda

当前推荐方案：

- **WSL**：作为主要本地开发环境
- **Node.js**：使用 `nvm`
- **Python**：使用 `uv`
- **本地基础服务**：按需使用 Docker

---

## 为什么这样做

这个项目更像是：

- 前端 / Web 应用
- 配套一个 Python agent 服务

因此更适合按语言分别管理，而不是试图用一个统一的 conda 环境包住全部内容。

当前阶段，`uv` 比 conda 更适合 Python 服务初始化；  
Node 侧则应使用 Node 自己的常见工具链。

---

## 环境管理建议

### 1. 不建议默认使用 conda

除非后续明确需要以下能力，否则先不要引入 conda：

- 本地重型模型
- CUDA / GPU 依赖
- 音频分析或训练相关的复杂科学计算环境
- 大量底层二进制依赖难以通过常规方式管理

如果当前主要是：

- Web 前端开发
- LLM API 调用
- agent 编排
- 常规 Python 服务

那么 **`uv` 就够了**。

### 2. Node 侧

Node.js 相关内容建议单独管理：

- 使用 `nvm` 管理 Node 版本
- 使用常规 JS/TS 包管理方式管理依赖

### 3. Python 侧

Python agent 服务建议：

- 使用 `uv`
- 保持独立虚拟环境
- 以 Python 服务的方式存在，不和前端混在一起

### 4. 本地依赖服务

如果后续需要数据库、缓存、向量库或其他基础设施，优先考虑：

- Docker
- Docker Compose

但当前阶段不需要为了“未来可能会用”而一次性搭很重的基础设施。

---

## 推荐的项目结构

当前只建议保留一个很轻的结构约束，细节由后续 opencode 决定：

```text
project-root/
  web/
  agent/
  infra/
```

含义：

- `web/`：前端或 Web 应用
- `agent/`：Python agent 服务
- `infra/`：可选的本地基础设施配置

不要求当前就把这三个目录全部做满，但建议从这个方向开始。

---

## 初始化原则

后续初始化时建议遵循以下原则：

### 1. 先轻量启动

先把开发环境和基础目录跑通，不要一开始引入过多框架和基础设施。

### 2. 前后端分开

Web 和 Python agent 不要混成一个环境。

### 3. 只解决当前问题

当前优先解决：

- 本地能开发
- 项目能启动
- 基础结构清晰

先不要提前设计太多产品层能力。

### 4. 保留后续升级空间

后面如果真的需要本地模型、训练、复杂音频处理，再单独引入 conda 或其他更重的方案。

## 给后续 opencode 的直接指令

后续请按以下方向初始化项目：

1. 以 **WSL** 作为主要开发环境
2. 将项目按 **Web / Agent / Infra** 的方向拆分
3. Web 侧使用 Node 工具链管理
4. Python 侧使用 **`uv`**
5. 当前**不要默认引入 conda**
6. 除非后续明确需要重型本地模型或复杂科学计算依赖，否则不要把初始化复杂化

## 一句话结论

当前阶段的初始化方案是：

> **WSL + nvm + uv +（按需）Docker，先不要用 conda。**
