# AI-Native Web Music Player：项目方向说明

## 1. 项目定位

本项目是一个 **AI-native 的 Web 音乐播放器**，目标不是简单复刻传统播放器，而是基于大模型与 agent 能力，探索一种从“被动搜索”走向“意图共鸣”的听歌体验。

它应当是一个：

- 面向 Web 的第三方音乐播放器
- 具备自然语言交互能力的音乐产品
- 能理解用户意图、情绪、场景与上下文的 agent 项目
- 后续可逐步扩展为“听歌助手 / 音乐陪伴型产品”

---

## 2. 核心目标

后续实现应围绕以下三个核心目标展开：

### 2.1 可用的基础播放器能力
首先保证它是一个可以正常使用的 Web 音乐播放器，而不是只有 AI 壳子的 demo。

包括但不限于：

- 基本播放控制
- 歌曲检索与播放
- 播放列表管理
- 基础用户状态与会话管理

### 2.2 AI 驱动的音乐理解与推荐
区别于传统关键词搜索，系统应能理解用户的模糊表达、情绪表达、场景表达与偏好表达。

例如：

- “我现在想听一点深夜开车的歌”
- “推荐一些不吵但也别太丧的中文歌”
- “我想从这首歌继续扩展一条风格接近但更高级的歌单”

### 2.3 Agent 化的交互体验
系统不只是返回搜索结果，而应作为一个音乐 agent 去完成目标。

理想状态下，它需要逐步具备：

- 识别用户意图
- 拆解任务
- 调用合适的音乐数据 / 检索 / 推荐能力
- 根据上下文持续调整结果
- 在多轮对话中保持一致的产品体验

---

## 3. 产品思路

后续设计时，建议将产品分成两层：

### 3.1 播放器层
这一层负责传统音乐产品的基础能力，是所有体验的承载底座。

关注点：

- 播放体验是否稳定
- 搜索、播放、切歌、列表等交互是否顺畅
- 页面结构是否足够清晰，适合后续插入 AI 能力

### 3.2 智能交互层
这一层负责 AI / agent 能力，是产品差异化的核心。

关注点：

- 如何接收自然语言输入
- 如何理解“意图、情绪、场景、偏好”
- 如何把用户请求转为播放器可执行的动作
- 如何给出可解释、可继续追问的结果

---

## 4. 技术路线建议

本项目建议采用 **前后端分层 + agent 独立服务** 的思路，而不是把所有逻辑揉在一起。

### 建议的总体结构

- `web/`：Web 前端，负责页面、播放器交互、聊天界面、状态展示
- `agent/`：Python 服务，负责 LLM、工具调用、推荐逻辑、任务编排
- `infra/`：本地开发与部署相关配置

### 环境管理建议

当前阶段建议采用：

- Node.js 侧使用 `nvm`
- Python 侧使用 `uv`
- 本地依赖服务使用 Docker

不建议一开始就把整个项目建立在 conda 之上。  
只有在后续引入重型本地模型、音频分析、训练或 CUDA 相关依赖时，再单独考虑 conda。

---

## 5. 推荐的开发原则

后续 opencode 在推进实现时，建议优先遵循以下原则：

### 5.1 先做最小闭环
优先完成一个“能听歌 + 能自然语言找歌 / 推荐歌”的最小版本，再逐步增加复杂能力。

### 5.2 先保证结构清晰，再追求功能丰富
前端、agent、数据源适配层、播放器能力应尽量分开，避免早期耦合过重。

### 5.3 AI 能力要可降级
即使 AI 能力暂时不稳定，基础播放器仍应可用，避免整个产品完全依赖大模型单点成功。

### 5.4 交互体验优先于模型炫技
重点不是“调用了多少 agent 框架”，而是用户是否真的更容易找到想听的音乐。

### 5.5 保留后续扩展空间
后续可能会加入：

- 长期偏好记忆
- 多轮推荐上下文
- 情绪 / 场景标签体系
- 歌单自动生成
- 音乐陪伴式对话
- 更复杂的推荐与排序策略

因此当前方案不必一次性做满，但应保留演进空间。

---

## 6. 第一阶段建议目标

第一阶段不追求完整产品，而是验证方向。

建议以以下结果为阶段性目标：

- 有一个可运行的 Web 播放器界面
- 能接入至少一套可用的第三方音乐数据来源
- 用户可以通过自然语言提出听歌需求
- agent 能返回歌曲、列表或推荐结果
- 用户能从推荐结果直接进入播放流程

只要这个闭环成立，后续再逐步扩展推荐质量、交互深度和产品完成度。

---

## 7. 非目标与边界

为了避免前期发散，当前阶段不建议过早深入以下方向：

- 复杂训练系统
- 过重的本地模型部署
- 过早优化多 agent 协作
- 过于细碎的工程基础设施
- 大而全的推荐系统重建

当前更重要的是验证：  
**“AI + 音乐播放器”是否真的能带来更好的听歌入口与体验。**

---

## 8. 对后续 opencode 的要求

后续实现时，请把这份文档视为“方向约束”，而不是“细节规范”。

意味着：

- 可以自主决定具体技术选型
- 可以自主拆分模块与实现细节
- 可以根据工程可行性调整架构
- 但不要偏离本项目的核心定位：  
  **这是一个以自然语言意图理解和 agent 体验为核心的 Web 音乐播放器，而不是普通播放器加一个聊天框。**

---

## 9. 一句话总结

这个项目的本质是：

> 用大模型与 agent 重构 Web 音乐播放器，让“找歌、听歌、发现音乐”的入口从操作驱动，转向意图驱动。
